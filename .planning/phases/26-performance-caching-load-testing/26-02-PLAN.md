  <action>
Create `src/app/api/monitoring/route.ts` as a GET endpoint that returns real-time infrastructure metrics.

The response shape must be:

```typescript
interface MonitoringResponse {
  status: 'ok' | 'degraded'
  timestamp: string           // ISO 8601
  realtimeConnections: number // supabase.getChannels().length — active client channels on THIS server instance
  avgQueryLatencyMs: number   // measured by timing a lightweight SELECT probe
  dbStatus: 'up' | 'down' | 'unconfigured'
}
```

Implementation:

1. **Realtime connection count**: Call `supabase.getChannels().length`. Note in a comment that this reflects channels on the current Node.js instance, not the global Supabase server. Return 0 if supabase is null.

2. **DB query latency**: Measure average of 3 consecutive `SELECT 1` probes via Supabase. Use `Date.now()` before and after. Average the 3 durations. This gives a real end-to-end latency including connection overhead.
   ```typescript
   const probes = 3
   const times: number[] = []
   for (let i = 0; i < probes; i++) {
     const t0 = Date.now()
     await supabase.from('drafts').select('id').limit(1)
     times.push(Date.now() - t0)
   }
   const avgQueryLatencyMs = Math.round(times.reduce((a, b) => a + b, 0) / times.length)
   ```

3. **Status logic**: If supabase is null, return `{ status: 'degraded', dbStatus: 'unconfigured', realtimeConnections: 0, avgQueryLatencyMs: -1 }`. If any probe throws, set `dbStatus: 'down'` and `status: 'degraded'`.

4. **Response headers**: Add `Cache-Control: no-store` — monitoring data must never be cached.

5. **HTTP status**: Return 200 for both `ok` and `degraded` states (monitoring consumers check `status` field, not HTTP code). Return 503 only if the endpoint itself throws an unhandled error.

6. Do NOT add authentication. This endpoint exposes only infrastructure metrics (no user data, no business logic, no secrets).

7. Export this type from the file for downstream use:
   ```typescript
   export type { MonitoringResponse }
   ```
  </action>
